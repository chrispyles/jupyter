This is a project that I completed as a part of my application for an internship at Two Six Capital. This notebook contains the original work that I did, but you can view the finished product, as well as the detailed instructions, on this project's GitHub page. The basis of the challange was to use a provided data set over transactions from one of Two Six Capital's businesses and answer four questions about the business:

  • Is the business improving or deteriorating?
  • Are there regional differences in the business?
  • Is there seasonality in customer behavior?
  • Is customer quality getting better or worse over time?

A Note About the Data Set¶

Some aspects of the data set are skewed and so conclusions can't be drawn based on them. For example, the data set has a row for each transaction that took place from Jan 2013 to Apr 2016. However, it includes a join_date column. But what if someone joined before Jan 2013 and had no transactions during the interval of the data? It is important that we do not let survivor bias cloud the results, so I will ignore the join_date column for most of the analysis.

def repeat(x, n):
    return [x for i in range(n)]

def not_nan(string):
    return string != 'nan'

def clean_date(string):
    date = string.split('/')
    if int(date[2]) > 18:
        year = '19' + date[2]
    else:
        year = '20' + date[2]
    if len(date[0]) == 1:
        month = '0' + date[0]
    else:
        month = date[0]
    if len(date[1]) == 1:
        day = '0' + date[1]
    else:
        day = date[1]
    return int(year + month + day)

def clean_table_dates(table, column):
    cleaned_dates = table.apply(clean_date, column)
    return table.drop(column).with_column(column, cleaned_dates)

def clean_data():
    # exclude where transaction date is 'nan'
    data1 = data.where('transaction_date', not_nan)
    tx_dates = clean_table_dates(data1, 'transaction_date')
    
    # exclude where join date is 'nan'
    tx_dates_not_nan = tx_dates.where('join_date', not_nan)
    all_dates_clean = clean_table_dates(tx_dates_not_nan, 'join_date')
    
    return all_dates_clean

3. A/B Test on Different Regions¶

In order to answer the second question, I decided to construct the framework to run A/B tests on each pair of regions to determine whether or not regionality exists in the data. My null and alternative hypotheses were:

  • Null Hypothesis: There is no difference between the two regions. Any difference in the observed data is due to random chance.
  • Alternative Hypothesis: Regionality exists in the data; that is, differences between regions are due to some factor other than random chance.

I decided to use the absolute difference between the mean number of transactions per year between the two regions as my test statistic. Here is the basic algorithm for the A/B test, which spans several code cells below:

  1. Choose two regions, R1 and R2, and then select the rows of the cleaned_data table where the region is R1 or R2.
  2. Randomly assign either R1 or R2 to each of the rows, regardless of their original designation. This means that the number of rows stays the same, but the ratio of R1 to R2 rows changes.
  3. Compute the mean number of transactions per year by dividing the number of rows for R1 and R2 by 4.
  4. Find the absolute different between the mean for R1 and R2
  5. Run steps 2-4 over and over, collecting the result of the shuffle in an array.
  6. Find the proportion of results in the array that are greater than or equal to the value of the test statistic for the original distribution. This is the p-value.
  7. If the p-value is 0.05 or less, conclude in the direction of the alternative hypothesis. Otherwise, conclude in the direction of the null hypothesis.

Here is some information from the docstrings on the functions defined below:

  • avg_counts: Groups rows of table by region and finds the average number of transactions per region per year; output is a table with columns 'region' and 'avg'
  • compute_ts_for_regions: Computes the test statistic for two regions in the cleaned_data table, either with shuffling or without (in the case of the original test statistic)
  • ab_test: Performs an A/B Test on two regions and returns the p-value
  • run_ab_tests: Runs the A/B Test on all possible combinations of regions A thru U

def avg_counts(table, column):
    grouped = table.group(column)
    ts_for_region = grouped.apply(lambda x: x/4, 'count')
    return grouped.drop('count').with_column('avg', ts_for_region)

def compute_ts_for_regions(r1, r2, shuffle=True):
    if shuffle:
        regions = cleaned_data.where('region', are.contained_in(make_array(r1, r2))).select('region')
        shuffled = np.random.choice(make_array(r1, r2), regions.num_rows)
        regions = regions.with_column('shuffled_region', shuffled).drop('region')

        # compute test statistics
        t = avg_counts(regions, 'shuffled_region')

        shuffled_dif = abs(t.column('avg').item(0) - t.column('avg').item(1))

        return shuffled_dif
    else:
        regions = cleaned_data.where('region', are.contained_in(make_array(r1, r2))).select('region')
        
        # compute test statistics
        t = avg_counts(regions, 'region')

        original_dif = abs(t.column('avg').item(0) - t.column('avg').item(1))

        return original_dif

def ab_test(r1, r2, reps):
    test_stats = make_array()
    original_ts = compute_ts_for_regions(r1, r2, shuffle=False)
    
    for i in np.arange(reps):
        stat = compute_ts_for_regions(r1, r2)
        test_stats = np.append(test_stats, stat)
    
    return np.count_nonzero(test_stats >= original_ts) / len(test_stats)

def run_ab_tests(reps):

    results = []
    for pair in pairs:
        p_val = ab_test(pair[0], pair[1], reps)
        results += [[pair, p_val]]
    
    tbl = Table().with_columns(
        'Region 1', [i[0][0] for i in results],
        'Region 2', [i[0][1] for i in results],
        'P-Value', [i[1] for i in results])
    return tbl

4. Finding an Alternative to run_ab_tests¶

Because the run_ab_tests function takes too long to run on my computer, I decided to select some regions that would give good information without running the A/B test on all 210 combinations. The functions define below have the ultimate effect of choosing 18 region pairs (6 with a low (< 100) value of the test statistic, 6 with a medium (between 100 and 1000) value, and 6 with a high (> 1000) value) and computing the p-values for those pairs of regions.

Here is some information on the functions below from their docstrings:

  • find_original_ts: Returns a table with each possible pair of regions and the value of the test statistic for that pair; returns a table with three columns: 'Region 1', 'Region 2', 'Test Statistic'
  • select_hi_mid_low: Selects pairs of regions based on whether or not their original test statistic value is high or low on a logarithmic scale
  • run_select_ab_tests: Runs the A/B Test on the pairs of regions chosen in the select_hi_mid_low function; returns a table with five columns: 'Region 1', 'Region 2', 'Original Test Stat', 'Category', 'P-Value'

        'Test Statistic', [i[1] for i in results])
    return tbl

def select_hi_mid_low():
    low = original_test_stats.where('Test Statistic', are.below(100)).sample(6)
    mid = original_test_stats.where('Test Statistic', are.between(100, 1000)).sample(6)
    hi = original_test_stats.where('Test Statistic', are.above(1000)).sample(6)
    
    sets_of_pairs = []
    for i in [low, mid, hi]:
        for j in np.arange(6):
            sets_of_pairs += [[i.row(j)[0], i.row(j)[1]]]
            
    return sets_of_pairs[0:7], sets_of_pairs[7:13], sets_of_pairs[13:]
        

low_ts_pairs, mid_ts_pairs, hi_ts_pairs = select_hi_mid_low()

def run_select_ab_tests():
    results = []
    for i in low_ts_pairs:
        result = ab_test(i[0], i[1], 1000)
        results += [[i, result]]
        
    for i in mid_ts_pairs:
        result = ab_test(i[0], i[1], 1000)
        results += [[i, result]]
        
    for i in hi_ts_pairs:
        result = ab_test(i[0], i[1], 1000)
        results += [[i, result]]
        
    original_stats = [original_test_stats.where('Region 1', i[0][0]).where('Region 2', i[0][1]).column('Test Statistic').item(0) for i in results]
    
    t = Table().with_columns(
        'Region 1', [i[0][0] for i in results],
        'Region 2', [i[0][1] for i in results],
        'Original Test Stat', original_stats,
        'Category', repeat('Low', 6) + repeat('Medium', 6) + repeat('High', 6), 
        'P-Value', [i[1] for i in results])
    return t

5. Determining Seasonality in Customer Behavior¶

To start answering the third question, I defined some useful functions that convert the integer YYYYMMDD dates into strings with the season and year. I exported two tables, seasons_by_year and users as csv files to make graphics in Sheets. The seasons_by_year table has the transaction dates replaced with season-year strings (e.g. 20181018 would become 'Fall 2018'). The users table has the transaction dates replaced with month-year strings (e.g. 20181018 would become 'Oct 2018').

def get_month(d):
    d = d//100
    d = d % 100
    return d

def month_to_string(d):
    d = get_month(d)
    months = ['Jan', 'Feb', 'Mar', 'Apr', 'May', 'Jun',
             'Jul', 'Aug', 'Sep', 'Oct', 'Nov', 'Dec']
    return months[d-1]

def get_year(d):
    return d // 10000

def get_season(d):
    month = get_month(d)
    if month in [12, 1, 2]:
        return 'Winter'
    elif month in [3, 4, 5]:
        return 'Spring'
    elif month in [6, 7, 8]:
        return 'Summer'
    elif month in [9, 10, 11]:
        return 'Fall'

def txn_date_to_season():
    seasons = cleaned_data.apply(get_season, 'transaction_date')
    years = cleaned_data.apply(lambda x: x//10000, 'transaction_date')
    data1 = cleaned_data.drop('transaction_date').with_column('season', seasons)
    data1 = data1.with_column('year', years)
    return data1

6. Business Deterioration¶

You'll notice when reading my report that the main argument I make is my answer to the first question: Is the business deteriorating? My answer was yes. All of the graphics I made in Sheets supported this trend, but I wanted to make sure that the regionality of the transaction volume wasn't making a trend appear out of thin air. In order to test this, I separated the transactions by region and created a plot of transaction volume for each region. As you can see below, the only region that had a net increase in transaction volume over the period was region J.

Conclusions¶

I concluded in my report that the business was deteriorating, but that there was a slight glimmer of hope. I will leave off with a quote from my report, summarizing what I thought of the trends:

The slight rise in transaction volume and gross revenue between Winter 2016 and Spring 2016 could be an indication that the business might be beginning to right itself, especially given the fact that there is incomplete data for Spring 2016, since more data could only raise the transaction volume and would also likely raise the gross revenue. Depending on how risky an investment the firm is willing to tolerate, this business could potentially bring itself back to the level it was at previously, if it manages to maintain the (albeit short) trend that might be beginning to arise in Spring 2016.
